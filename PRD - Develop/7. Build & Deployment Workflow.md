## 7. Build & Deployment Workflow

1. **Development Lokal:** Jalankan `npm install` -> `npm run dev`. Gunakan mock data, karena `google.script.run` tidak berjalan di luar ekosistem GAS (sediakan mock layer di `gasClient.js`).
2. **Backend Modular:** Development backend wajib dilakukan secara modular di dalam folder `gas-src/` (per modul), **bukan mengedit langsung di file gabungan**.
3. **Build Backend (`npm run build:gas`):** Menjalankan script `scripts/build-gas.mjs` yang akan:
   * Menyalin `gas-src/00-setup.gs` menjadi `dist-gas/setup.gs` (tanpa modifikasi).
   * Menggabungkan (concat) `01-main.gs` + `02-auth.gs` + `modules/*.gs` (berdasarkan urutan prefix angka) + `99-utils.gs` menjadi `dist-gas/code.gs`, lengkap dengan header komentar penanda asal tiap blok kode.
4. **Build Frontend (`npm run build`):** Menjalankan Vite build menggunakan `vite-plugin-singlefile` untuk menghasilkan 1 file HTML di folder `dist/`, kemudian disalin/di-rename menjadi `webapp.html`.
5. **Build Keseluruhan (`npm run build:all`):** (Direkomendasikan) Script gabungan untuk mengeksekusi langkah 3 & 4 secara bersamaan.
6. **Push ke Apps Script:** Salin `dist/webapp.html`, `dist-gas/code.gs`, dan `dist-gas/setup.gs` ke project Apps Script. (Sangat direkomendasikan menggunakan `clasp push` untuk standar enterprise agar versioning Git terjaga; arahkan `rootDir` di `.clasp.json` ke folder output tersebut).
7. **Inisialisasi Database:** Jalankan fungsi `runSetup()` satu kali secara manual dari editor GAS untuk melakukan inisialisasi skema Sheets.
8. **Deployment Web App:** Deploy project sebagai **Web App** (Konfigurasi: `Execute as: Me`, `Who has access: sesuai kebutuhan organisasi`).
9. **Automasi CI/CD:** Manfaatkan `clasp`, `.clasp.json`, dan `appsscript.json` (manifest: set `webapp.executeAs` & `access`) agar proses deployment ke GAS dapat diotomatisasi melalui workflow `.yml` (serupa dengan `deploy-pages.yml` di 4.3, namun dengan target Apps Script).

> **Ringkasan package.json scripts:** `dev` (lokal), `build` (frontend/Vite), `build:gas` (backend concat), `build:all` (frontend + backend), `build:pages` (mode preview untuk GitHub Pages).

### 7.1. Dokumentasi Resolusi Bug & Aturan Optimasi Kode (PENTING)

Untuk menjaga stabilitas eksekusi frontend di dalam Google Apps Script (GAS) dan menghindari masalah layar putih kosong (blank screen) tanpa pesan error, tim pengembang wajib mengikuti aturan teknis berikut:

#### A. Akar Permasalahan (Root Cause Analysis)
1. **Kegagalan Pemisahan Baris Kode (Arbitrary Line Splitting):**
   * **Penyebab:** Pada konfigurasi awal, skrip pasca-proses `/scripts/build-gas.mjs` menggunakan fungsi `splitLineSafely` untuk memecah baris yang melebihi 300 karakter demi mematuhi batas panjang baris hitoris editor GAS.
   * **Masalah:** Ketika Vite melakukan minifikasi default, seluruh kode JavaScript dikompresi menjadi satu baris HTML/JS yang sangat panjang. Fungsi `splitLineSafely` memotong baris tersebut secara semi-acak (di dekat koma `,` atau spasi ` `), yang sering kali memotong tepat di tengah-tengah ekspresi JavaScript (seperti string literal, nama variabel, atau pemanggilan fungsi). Hal ini mengakibatkan kegagalan penafsiran sintaksis (`SyntaxError: Invalid or unexpected token`) saat browser memuat iframe GAS, sehingga merusak seluruh inisialisasi aplikasi.
2. **Tabrakan Instance React (Dual React Instance):**
   * **Penyebab:** Pustaka pihak ketiga seperti `lucide-react` dan `recharts` yang dimuat dari CDN `esm.sh` secara default mengunduh dan menginstansiasi salinan React mereka sendiri di latar belakang.
   * **Masalah:** Adanya dua instance React aktif yang berjalan bersamaan di satu window memicu tabrakan context, hook error, dan kegagalan rendering UI yang berujung pada layar blank putih.

#### B. Solusi Permanen & Konfigurasi yang Wajib Dipertahankan
1. **Gunakan Terser Minifier yang Terstruktur:**
   * Di dalam `vite.config.ts`, build target wajib menggunakan Terser minifier (`minify: 'terser'`). Terser mengompresi kode JavaScript dengan mempertahankan integritas token sintaksis dan struktur kode, sehingga aman dari kerusakan logika.
2. **Hapus Pemotong Baris Buta di Build Script:**
   * Skrip `/scripts/build-gas.mjs` **dilarang keras** menggunakan fungsi pemisah baris berbasis batas karakter statis (seperti `splitLineSafely`). Sebagai gantinya, gunakan pembersihan whitespace terarah (seperti fungsi `formatHtmlForGas`) yang hanya menghapus spasi redundan di antara tag HTML (`html.replace(/>\s+</g, '><')`) dan menyerahkan optimalisasi pemampatan naskah JS sepenuhnya kepada Terser.
3. **Isolasi Dependensi via Import Maps:**
   * Di file `/index.html`, semua library pendukung yang bergantung pada React wajib dipetakan dengan parameter kueri khusus `?external=react,react-dom` (contoh: `https://esm.sh/lucide-react@0.460.0?external=react,react-dom`).
   * Di dalam `vite.config.ts`, deklarasikan library tersebut sebagai `external` dalam konfigurasi `rollupOptions.external` agar tidak dibundle ganda oleh Vite.
4. **Penguatan Keamanan Sandbox Iframe (`01-main.gs`):**
   * Di dalam backend GAS, `HtmlOutput` yang dikembangkan oleh `doGet()` wajib mengaktifkan `.setSandboxMode(HtmlService.SandboxMode.NATIVE)` dan menyematkan meta tag `X-UA-Compatible` bernilai `IE=Edge` untuk memastikan kompatibilitas serta memperketat keamanan dari risiko celah keamanan sandbox escape di lingkungan Google Workspace.
