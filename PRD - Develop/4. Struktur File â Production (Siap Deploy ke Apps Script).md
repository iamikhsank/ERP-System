## 4\. Struktur File — Production (Siap Deploy ke Apps Script)

Hanya **3 file** (hasil `dist-gas/` + `dist/`) yang di-upload ke Apps Script Editor:\[cite: 2]

|File|Isi|Sumber|
|-|-|-|
|`setup.gs`|One-time setup: membuat struktur Sheets, header kolom tiap modul, seed data awal (role, user admin default), inisialisasi Script Properties (config/env). Dijalankan manual sekali via fungsi `runSetup()`.|Copy langsung dari `gas-src/00-setup.gs` (tidak perlu di-concat, memang sudah 1 file).\[cite: 2]|
|`code.gs`|Seluruh backend logic hasil gabungan: `doGet(e)` untuk render `webapp.html`, semua fungsi CRUD per modul, auth/session check, routing API internal.|**Hasil build otomatis** (`npm run build:gas`) dari `scripts/build-gas.mjs` yang meng-concat `01-main.gs` + `02-auth.gs` + seluruh `modules/\*.gs` + `99-utils.gs` sesuai urutan prefix angka, ditambah header komentar penanda asal tiap blok (mis. `// ==== inventory.gs ====`) supaya tetap mudah ditelusuri walau sudah jadi satu file.\[cite: 2]|
|`webapp.html`|Hasil **build Vite** — single file HTML dengan seluruh JS \& CSS aplikasi ter-inline \& ter-minify (library tetap via CDN, lihat 4.1), dipanggil oleh `doGet()` via `HtmlService.createHtmlOutputFromFile('webapp')`.|Hasil `npm run build` (Vite + `vite-plugin-singlefile`).\[cite: 2]|

**Catatan penting soal `index.html` vs `webapp.html`:**\[cite: 2]

* `index.html` = versi RAW (multi-file, readable, untuk development/debugging lokal via Vite dev server) — **tidak diupload ke GAS**.\[cite: 2]
* `webapp.html` = versi BUILD (single-file hasil `vite build` with plugin single-file) — **inilah yang diupload ke GAS** dan dipanggil oleh `code.gs`.\[cite: 2]

**Alur kerja penulisan backend:** developer/AI **tidak pernah** mengedit `dist-gas/code.gs` secara manual — semua perubahan logic dilakukan di file modular `gas-src/modules/\*.gs`, lalu jalankan `npm run build:gas` untuk regenerate `dist-gas/code.gs`.\[cite: 2] Ini paralel dengan cara frontend bekerja (edit di `src/`, build ke `webapp.html`).\[cite: 2]

### 4.1 Wajib Pakai CDN + Import Map

**Alasan:** menjaga `webapp.html` tetap kecil (karena ada batas ukuran \& performa load `HtmlService` di GAS), mempercepat build, dan menghindari duplikasi library di dalam bundle.\[cite: 2]

**Aturan:**

* Semua library pihak ketiga (Chart.js/ApexCharts, date library, icon set, dsb) **dilarang** di-`npm install` lalu dibundle Vite. Load via CDN (`jsDelivr` atau `esm.sh`, versi selalu dikunci/pin, tidak `@latest`).\[cite: 2]
* Gunakan native browser **Import Map** di `<head>` (baik di `index.html` raw maupun sebagai hasil akhir di `webapp.html`) supaya `import` statement di kode aplikasi tetap bersih tanpa perlu bundler resolve library.\[cite: 2]
* Tailwind CSS dipasang lewat CDN script (Play CDN) untuk styling cepat, atau via CDN precompiled CSS jika butuh konsistensi produksi lebih ketat.\[cite: 2]
* Font di-load via Google Fonts CDN (`<link rel="preconnect">` + `<link rel="stylesheet">`), bukan self-hosted, agar file tetap ringan.\[cite: 2]

**Contoh Import Map (ditaruh di `<head>` sebelum script module utama):**\[cite: 2]

```html
<script type="importmap">
{
  "imports": {
    "chart.js": "\[https://esm.sh/chart.js@4](https://esm.sh/chart.js@4)",
    "date-fns": "\[https://esm.sh/date-fns@3](https://esm.sh/date-fns@3)",
    "lucide": "\[https://esm.sh/lucide@latest](https://esm.sh/lucide@latest)"
  }
}
</script>
```

**Contoh pemakaian di kode aplikasi (tanpa perlu di-bundle Vite):**\[cite: 2]

```js
import { Chart } from "chart.js";
import { format } from "date-fns";
```

* Vite dikonfigurasi agar **tidak** mem-bundle module yang sudah ada di import map (`build.rollupOptions.external`), sehingga hasil `webapp.html` hanya berisi kode aplikasi sendiri + referensi CDN.\[cite: 2]

### 4.2 Error Handling Frontend (Production-Grade Console UI)

Wajib ada sistem penangkap error di sisi frontend yang **tidak menampilkan error mentah browser** ke user, melainkan panel khusus yang rapi dan profesional:\[cite: 2]

* **Global error boundary:** pasang `window.onerror` dan `window.addEventListener('unhandledrejection', ...)` di `main.js` untuk menangkap seluruh runtime error \& promise rejection yang tidak tertangani.\[cite: 2]
* **Error dari backend GAS:** setiap pemanggilan `google.script.run` yang gagal (`withFailureHandler`) diarahkan ke handler terpusat yang sama, bukan `alert()` bawaan browser.\[cite: 2]
* **Komponen `errorConsole.js`:** panel overlay (bisa di-toggle, posisi bottom-right seperti devtools console mini) yang menampilkan:\[cite: 2]

  * Daftar error dengan timestamp, severity (`error`/`warning`/`info`), pesan singkat untuk user awam, dan detail teknis yang bisa di-*expand* (stack trace, response GAS) — dikumpulkan agar dev tetap bisa debug tanpa mengganggu user biasa.\[cite: 2]
  * Badge counter jumlah error di topbar (mirip notifikasi), agar admin/tech-savvy user sadar ada error tanpa menghentikan alur kerja user biasa.\[cite: 2]
  * Tombol "Copy error detail" untuk dilaporkan ke tim IT/dev.\[cite: 2]
  * Untuk user role non-admin, tampilkan versi ringkas saja ("Terjadi kesalahan, silakan coba lagi atau hubungi admin") — detail teknis hanya terbuka untuk role `admin`.\[cite: 2]
* Semua error tetap dikirim ke `console.error` asli browser juga (tidak menutup akses devtools normal), panel ini hanya lapisan tambahan yang lebih presentable untuk end user \& mempermudah triase.\[cite: 2]
* Gaya visual konsisten dengan design system (dark surface, warna semantik: merah untuk error, kuning untuk warning), bukan `alert()`/`confirm()` bawaan browser sama sekali di seluruh aplikasi.\[cite: 2]

### 4.3 Deployment Preview via GitHub Pages

Karena `google.script.run` tidak bisa jalan di luar environment GAS, GitHub Pages dipakai sebagai **preview UI statis** (dengan mock data) — bukan pengganti deployment utama ke Apps Script.\[cite: 2]

`.github/workflows/deploy-pages.yml`:\[cite: 2]

```yaml
name: Deploy Preview to GitHub Pages

on:
  push:
    branches: \[ "main" ]
  workflow\_dispatch:

permissions:
  contents: read
  pages: write
  id-token: write

concurrency:
  group: "pages"
  cancel-in-progress: true

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Build (mode preview, mock data GAS)
        run: npm run build -- --mode preview

      - name: Setup Pages
        uses: actions/configure-pages@v5

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./dist

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page\_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

**Catatan:** mode `preview` di `vite.config.js` mengaktifkan `gasClient.js` versi mock (data dummy in-memory) sehingga UI tetap bisa dibuka \& didemokan tanpa backend GAS asli.\[cite: 2]
